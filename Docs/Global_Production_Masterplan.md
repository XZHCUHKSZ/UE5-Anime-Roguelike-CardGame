# 全局研发总计划（AnimeRogue）

## 0. 项目总目标
- 玩法框架：对齐《杀戮尖塔》的单局构筑循环
- 内容风格：二次元原创角色、怪物、事件、美术与文本
- 交付目标：先做可玩垂直切片，再做内容量产，最后性能优化与发版

## 1. 阶段路线图（建议 24 周）

### 阶段A：预制作（Week 1-3）
目标：把“做什么”锁死，防止后续返工。
- 锁定核心机制清单：抽牌、能量、回合、地图、奖励、商店、事件、遗物
- 锁定技术架构：`BattleManager + RunManager + DataTable + SaveGame`
- 锁定美术风格与UI方向（角色、敌人、节点、卡面）
- 输出物：
  - 全局GDD（本仓库文档体系）
  - 技术文档 v1
  - 可编译工程骨架

### 阶段B：垂直切片（Week 4-8）
目标：做出“能玩且像样”的一条完整线路。
- 完整链路：`StartRun -> Battle -> Reward -> Event/Shop/Rest -> Boss`
- 内容规模（切片版）：
  - 1角色
  - 20-30张卡
  - 6普通敌 + 1精英 + 1Boss
  - 8-10事件
- 光照/性能门槛（第一版）：
  - 1080p 开发机稳定 60 FPS
  - 不出现长时间卡顿（>100ms）
- 输出物：
  - 可试玩演示包
  - 第一轮数值反馈

### 阶段C：量产开发（Week 9-16）
目标：扩大内容并保持系统稳定。
- 扩展到目标MVP规模（2角色、60卡、12+2敌、25事件、40遗物）
- 卡牌效果执行器完善（按 `EffectScriptId`）
- 事件脚本系统完善（可复用参数）
- 中期性能治理：
  - UI刷新优化
  - 特效预算
  - 贴图与材质预算
- 输出物：
  - 内容冻结候选版（Feature Complete）

### 阶段D：优化与发版准备（Week 17-24）
目标：从“能玩”变成“能发布”。
- 性能优化：光照、后处理、阴影、粒子、UI、加载时间
- 稳定性优化：崩溃、坏档、边界条件
- QA与回归：功能回归、存档兼容、平衡回归
- 发版素材：商店图、说明、演示视频
- 输出物：
  - RC（Release Candidate）
  - 发布版构建

## 2. 每阶段硬验收门（Gate）
- Gate A（预制作结束）：核心玩法和技术路线不再频繁变更
- Gate B（切片结束）：完整单局可玩，不靠手工“补流程”
- Gate C（量产结束）：内容达标，系统不缺关键模块
- Gate D（发版结束）：性能与稳定性达到发布标准

## 3. 全局并行工作流
- 系统开发线：战斗/跑图/存档/工具链
- 内容生产线：卡牌/敌人/事件/遗物数据生产
- 体验打磨线：UI、音效、演出、引导
- 质量保障线：自动化检查 + 每周回归

## 4. 光照与性能策略（全程约束）
- 阶段B前：优先可玩，不追求最终画质
- 阶段C：开始性能预算管理（每场景统计）
- 阶段D：针对低端机档位做最终优化
- 原则：每个阶段都做轻量优化，不把性能债堆到最后
